# iptime N602SR UART 펌웨어 추출 프로젝트 — 종합 정리

**작성일**: 2026-07-24
**대상 장비**: ipTIME N602SR 공유기 (Realtek RTL8197F SoC)
**최종 상태**: 목표 달성 — UART `<RealTek>` 부트로더를 통해 8MB 플래시 전체를 자동화된 방식으로 안정적으로 추출하는 데 성공했고, `master`에 병합 완료.

---

## 1. 목표

시리얼(UART) 콘솔로 노출된 Realtek 2차 부트로더(`<RealTek>` 프롬프트)의 `FLR`(Flash Read, 플래시→RAM 복사)과 `DB`(RAM 덤프) 명령을 이용해, 분해 없이 라우터의 8MB SPI 플래시 전체를 PC로 추출하는 자동화 도구를 만드는 것.

## 3. Problem B: FLR confirm 자동 Abort 조사

### 3.1 현상
`FLR` 명령 실행 후 `(Y)es , (N)o ?` 확인 프롬프트가 뜨는데, PC가 아무것도 보내지 않았거나 `Y`를 보내도 부트로더가 곧바로 `Abort!`를 출력하며 명령을 취소하는 현상이 반복적으로 발생.

### 3.2 정적 분석 (2회, 8MB 레퍼런스 펌웨어의 2차 부트로더 디스어셈블 기반)
- **1차**: `uart_getchar_blocking()`이 순수 상태-비트 busy-wait이며 어디에도 타임아웃 코드가 없음을 확인 → "타임아웃 때문에 Abort"라는 가설(H2) 기각.
- **2차**: confirm 판정 루틴이 수신한 모든 바이트를 실제로 echo하며(출력 깨짐이 아님), CR/LF를 받거나 **최대 2바이트**를 채워야 반환됨을 확인. `read_confirm_char()`는 오직 `buf[0]`만 검사 — 즉 앞에 잡음 바이트가 하나라도 끼면 그 잡음이 `buf[0]`을 차지하고 실제 `Y`는 무시되는 `buf[1]`에 떨어짐.

### 3.3 결정적 실험 — 무송신(Silent Observation)
`flr_confirm_silent_observation()`: confirm 프롬프트 감지 후 PC가 **구조적으로 아무 것도 전송할 수 없는** 코드 경로에서 5초간 관찰. **5/5 전부** confirm 감지 후 6.4~8.6ms 만에 `Abort!` 발생 — PC가 아무것도 보내지 않았는데도 뭔가 수신됨. 첫 잡음 바이트는 매번 동일하게 `0xFE`, 유휴-high 라인의 순간적 글리치가 스타트비트로 오인되는 전형적인 UART 프레이밍 글리치 패턴과 일치.

추가로, 동일한 잡음 패턴이 confirm과 무관한 일반 명령(`?`)에서도 20회 중 19회 재현되어, "confirm 순간에만 발생하는 문제"가 아니라 **부트로더가 입력을 기다리기 시작할 때마다 발생하는 일반적 현상**임을 확인.

### 3.4 근본 원인 확정
사용자가 UART 각 배선을 서로 접촉/근접하지 않도록 물리적으로 고정한 뒤, FLR→confirm→Y→"Flash Read Successed!" 전 과정이 즉시 안정적으로 성공. 지금까지 모은 모든 증거(진짜 echo, 무송신 상태에서도 수신, confirm과 무관한 일반 발생)와 정확히 부합 — **인접 배선 간 전기적 커플링(crosstalk)이 원인**이었음이 사실상 확정.

## 4. FlashDumper 구현과 실기 디버깅 6라운드

Problem B가 해결되면서, 그동안 `NotImplementedError`로 막아두었던 2단계(`FlashDumper`)를 실제 구현으로 교체. TDD로 187개 테스트를 갖춘 뒤 실기 테스트를 시작했지만, 각 라운드마다 새로운 실기 전용 버그가 드러났다.

| 라운드 | 증상 | 실제 원인 | 수정 |
|---|---|---|---|
| 1 | 4청크 전부 "Unknown command" | 청크 사이에 `clean_prompt()`가 없어 다음 FLR의 선두 바이트가 유실됨 | 매 청크 시작 시 `clean_prompt()` 호출 (`c66a8b3`) |
| 2 | 4청크 전부 CLEANUP 단계에서 Abort | `resolve_timeout`(2.0s)이 16바이트짜리 테스트 데이터 기준값이라 실제 4096바이트 청크엔 부족 | `confirm_timeout`/`resolve_timeout`을 설정 가능하게 변경 (`68a74d6`) |
| 3 | 10초로 늘려도 동일 패턴 | 실제로는 느린 처리가 아니라 **영원히 무응답** — bare `Y`(CR 없음)로는 `line_input()`의 "2바이트 또는 CR/LF" 종료 조건을 절대 만족 못함. 배선 수정 전엔 잔여 노이즈가 우연히 "2번째 바이트" 역할을 해서 동작했던 것 | `Y` → `Y\r`로 변경, 사용자가 실기로 직접 검증 후 확정 (`6a4db59`) |
| 4 | FLR/confirm은 성공하지만 DB 응답이 항상 1000바이트 근처에서 끊김 | (오진단) "DB 명령 출력 버퍼 한계"로 오판 → sub-chunk 분할로 "수정" (`8ac51bd`, **이후 되돌림**) | — |
| 5 | 위 "수정"이 새로운 방식으로 깨짐 (`3E0` 길이 요청 시 3바이트만 반환) | 프로젝트 사상 최초로 명령 인자에 16진 문자(A-F)가 포함됨 → **실제 원인**: `DB` 명령의 length 인자는 `FLR`과 달리 **10진수로 파싱**됨. `commands.db()`가 계속 16진수 문자열을 보내고 있었던 것이 진짜 원인 (`DB 80100000 200` → 정확히 10진 200바이트, `DB 801003E0 10` → 정확히 10진 10바이트로 실기에서 직접 검증) | `commands.db()`를 10진수 포맷으로 수정, sub-chunk 로직 되돌림 (`273e4f9`) |
| 6 | — | 재시도 결과 4/4 청크 전부 성공, 실제 플래시 데이터와 바이트 일치 확인 | 프로젝트 사상 최초의 완전 자동, 무중단, 다중 청크 덤프 성공 |

## 5. 전체 8MB 덤프 및 신뢰성 검증

- `dump --offset 0x0 --length 0x800000` 실행 → **2048/2048 청크, 실패 0건**, 약 4.9시간 무인 실행(청크당 ~8.6초).
- 저장소에 있던 레퍼런스 펌웨어 파일(`n602sr_ml_15_258.bin`)과 비교했을 때 상당수 바이트가 달랐으나, 분석 결과 그 레퍼런스 파일 자체가 (지워진 플래시를 `0xFF`가 아닌 `0x00`으로 패딩하는) 업그레이드 패키지로 보이며, 실제 원시 플래시 덤프와의 1:1 비교 기준으로 부적절함이 드러남.
- 부트로더 stage2 gzip 스트림(offset `0x9a60`) 내부의 단일 바이트(`0x9b80`) 불일치가 압축 해제 실패를 유발했지만, **같은 영역을 독립적으로 두 번(오프셋 오타로 한 번, 정확한 오프셋으로 한 번 더) 재덤프해 바이트 단위로 완전히 동일한 결과**를 얻어 — 전송 노이즈가 아니라 이 장비의 실제 현재 펌웨어가 레퍼런스 파일과 다른 빌드임을 확인.
- **결론: `firmware_full.bin`(8,388,608바이트)은 재현 가능하고 신뢰할 수 있는 전체 플래시 덤프.**

## 8. 알려진 사소한 이슈 (미해결, 낮은 우선순위)

`FlashDumper.run()`은 청크를 `offset` 위치에 쓰지만 파일을 `flash_size`까지만 truncate하기 때문에, `--offset`이 `--length`보다 큰 경우(예: 특정 구간만 검증용으로 재덤프할 때) 파일 앞부분이 0으로 채워진 채로 생성됨. 항상 offset 0부터 전체를 덤프하는 실사용 시나리오에서는 무해하지만, 부분 구간만 별도로 검증할 때 파일이 요청 구간부터 시작하지 않는다는 점에 유의.
