# UART 추출 바이너리 vs 기존 레퍼런스 바이너리 비교 분석

**작성일**: 2026-07-24
**비교 대상**
- **UART 추출본**: `firmware_full.bin` — `uart_dumper` 자동화 도구로 `<RealTek>` 부트로더를 통해 실기에서 직접 덤프한 8MB 전체 SPI 플래시
- **레퍼런스 바이너리**: `n602sr_ml_15_258.bin` — 저장소에 이미 있던 기존 펌웨어 파일 (`firmware_extract/_n602sr_ml_15_258.bin.extracted/`에 squashfs까지 이미 추출되어 있어, 이전부터 정적 분석용으로 쓰여 온 파일)

**결론 먼저**: 두 파일은 상당수 바이트가 다르지만(약 1.35%), 그 차이는 UART 덤프의 결함이 아니라 **`n602sr_ml_15_258.bin`이 이 장비의 "원시 플래시 덤프"가 아니라 별도의 펌웨어 업그레이드 이미지(다른 빌드, 다른 패딩 방식)이기 때문**임을 여러 각도에서 확인했다. `firmware_full.bin`은 재현성 검증(같은 구간을 독립적으로 두 번 재덤프해 바이트 단위로 완전히 일치)까지 마친, 신뢰할 수 있는 실제 원시 플래시 덤프다.

## 2. 바이트 단위 비교 (겹치는 구간, 8,163,328바이트)

| 항목 | 값 |
|---|---|
| 일치 바이트 | 8,052,824 (98.646%) |
| 불일치 바이트 | 110,504 (1.354%) |

대부분은 일치하지만, 나머지를 어떻게 해석할지가 이 보고서의 핵심이다.

### 2.1 4096바이트 청크 단위 분포

| 분류 | 청크 수 | 비율 |
|---|---|---|
| 완전 일치 | 29 | 1.5% |
| 부분 불일치 | 1,948 | 97.7% |
| 완전 불일치 | 16 | 0.8% |

완전 불일치 16개 청크는 **연속된 하나의 구간**(offset `0x30000`–`0x40000`, 64KB)에 몰려 있다. "여기저기 흩어진 손상"이 아니라 특정한 한 블록이 통째로 다르다는 뜻이다.

### 2.2 불일치의 성격 — run(연속 불일치 구간) 길이 분포

| run 길이 | 발생 횟수 | 총 바이트 |
|---|---|---|
| 65,536 (하나의 블록) | 1 | 65,536 |
| 1 (고립된 단일 바이트) | 41,967 | 41,967 |
| 2 | 1,389 | 2,778 |
| 3 | 25 | 75 |
| 그 외 (4~56) | 소수 | 128 |

즉 전체 110,504바이트 불일치 중:
- **65,536바이트(59.3%)**는 offset `0x30000`–`0x40000`의 단일 블록 하나에서 발생
- 나머지 **44,968바이트(40.7%)**는 거의 전부가 1~2바이트짜리 고립된 산발적 차이(41,967개의 완전히 독립적인 1바이트 불일치 + 1,389개의 2바이트 불일치)

이 두 종류는 원인이 서로 다르다 — 아래에서 각각 설명한다.

## 3. 원인 1: offset `0x30000`–`0x40000` 블록 — "지워진 플래시" 표현 방식 차이

이 블록의 내용을 직접 비교하면:

```
ref[0x30000:0x30040] = 00 00 00 00 00 00 ... (전부 0x00)
new[0x30000:0x30040] = ff ff ff ff ff ff ... (전부 0xFF)
```

64KB 전체가 레퍼런스는 **전부 `0x00`**, UART 덤프는 **전부 `0xFF`**다. 이는 코드/데이터 내용이 다른 것이 아니라, **지워진(erased) 플래시 영역을 표현하는 방식의 차이**다:

- 실제 NOR/SPI 플래시 칩은 지워진 상태에서 **`0xFF`**로 읽힌다(하드웨어 특성) — UART 덤프가 보여주는 값이 물리적으로 맞다.
- 반면 `0x00`으로 채워진 것은 **펌웨어 이미지 빌드 도구가 미사용/예약 영역을 0으로 패딩**하는 전형적인 방식이다.

파일 전체에서 이 패턴을 집계하면:

| | 전부 `0xFF`인 청크 | 전부 `0x00`인 청크 |
|---|---|---|
| 레퍼런스 | 12개 | 29개 |
| UART 덤프 | 83개 | 12개 |

UART 덤프가 `0xFF`(진짜 지워진 플래시) 청크를 훨씬 많이 갖고 있고, 레퍼런스는 `0x00`(패딩) 청크가 더 많다 — **레퍼런스 파일이 원시 플래시 덤프가 아니라 이미지 빌드 산출물(업그레이드 패키지 등)일 가능성**을 뒷받침한다.

추가로, UART 덤프에만 존재하는 마지막 225,280바이트(레퍼런스 크기를 넘어서는 `0x7C9000`–`0x800000` 구간)는 **전부 `0xFF`**다 — 8MB 칩의 미사용 꼬리 영역이 그대로 지워진 채 남아있는, 실제 플래시라면 자연스러운 모습이다. 레퍼런스 파일은 이 구간 자체가 아예 없다(파일이 거기서 끝남).

## 4. 원인 2: 나머지 산발적 1~2바이트 차이 — 다른 빌드/버전

### 4.1 밀림(byte-shift) 여부 확인

가장 먼저 의심한 것은 "UART 덤프 파이프라인이 특정 지점에서 바이트를 하나 빠뜨리거나 중복시켜서, 그 이후 모든 내용이 밀려 보이는 것 아닌가"였다. 이는 심각한 도구 버그를 의미하므로 별도로 검증했다.

두 곳(서로 다른 오프셋)에서 ±4바이트 범위로 정렬을 흔들어 일치율을 재확인:

- offset `0x6E0` 근방: shift 0에서 4074/4096(99.5%) 일치, 다른 shift는 전부 300~1000대로 급락
- offset `0x5DAE50` 근방: shift 0에서 2037/2048(99.5%) 일치, 다른 shift는 전부 20 이하로 급락

**두 지점 모두 shift=0(밀림 없음)이 압도적으로 최선.** 즉 바이트가 밀리거나 중복된 흔적이 없다 — 도구 버그가 아니다.

### 4.2 결론: 서로 다른 펌웨어 빌드

밀림이 없다는 것은, 이 산발적 차이들이 **같은 위치에서 실제로 값이 다른 것**이라는 뜻이다. 재컴파일된 바이너리는 임베디드 타임스탬프/버전 문자열/주소 상수 등으로 인해 코드 대부분이 동일해도 곳곳에 소수 바이트씩 차이가 나는 것이 일반적이다 — 지금 관찰된 패턴(98.6% 일치 + 수만 개의 고립된 1~2바이트 차이)과 정확히 부합한다.

즉 **이 장비에 현재 실제로 올라가 있는 펌웨어와 `n602sr_ml_15_258.bin`은 서로 다른 버전/빌드**로 보인다.

## 5. 사례 연구: 부트로더 stage2 gzip 스트림(offset `0x9a60`)

이전 정적 분석(2026-07-22)에서 사용한 2차 부트로더 압축 스트림을 UART 덤프 기준으로 다시 열어보면:

```
레퍼런스: zlib.decompressobj(47).decompress(ref[0x9a60:]) → 성공, 101,264바이트
UART 덤프: zlib.decompressobj(47).decompress(new[0x9a60:]) → 실패
           Error -3: invalid distance too far back
```

두 파일을 바이트 단위로 비교하면 이 스트림 구간(`0x9a60` 이후 320바이트)에서 **단 1바이트**만 다르다:

```
offset 0x9b80: 레퍼런스=0xAE, UART 덤프=0xB0
```

이 한 바이트가 gzip 압축 해제 실패의 직접 원인이다(deflate는 비트스트림이라 한 바이트만 달라도 그 이후 전부 깨짐).

### 5.1 재현성 검증 — 전송 노이즈가 아님을 실기로 확인

이 바이트가 UART 전송 중 생긴 잡음인지, 아니면 실제로 장비에 그렇게 저장되어 있는 값인지 구분하기 위해, 같은 구간(`0x9800`–`0xA000`)을 **독립적인 별도 세션에서 다시 한번 UART로 재덤프**했다:

- 재덤프한 2048바이트 전체가 원본 `firmware_full.bin`의 같은 구간과 **바이트 단위로 완전히 일치**
- 문제의 offset `0x9b80`도 다시 읽었을 때 역시 `0xB0` — 두 번 모두 동일

두 번의 독립적인 실기 읽기가 완전히 같은 값을 냈다는 것은, 이 값이 **장비에 실제로 저장되어 있는 값**이지 무작위 전송 잡음이 아니라는 뜻이다. 따라서 이 gzip 압축 해제 실패는 UART 덤프의 결함이 아니라, **이 장비의 실제 stage2 부트로더 바이너리가 레퍼런스 파일의 것과 다르기 때문**이다.

## 6. 종합 결론

1. **`firmware_full.bin`(UART 덤프)은 신뢰할 수 있다.** 밀림/중복 없음(도구 버그 배제), 지워진 영역이 물리적으로 올바른 `0xFF`로 나타남, 의심 지점을 독립적으로 재읽기해도 완전히 재현됨.
2. **`n602sr_ml_15_258.bin`(레퍼런스)은 이 장비의 원시 플래시 1:1 사본이 아닐 가능성이 높다.** 크기가 8MB보다 작고, 미사용 영역을 `0xFF`가 아닌 `0x00`으로 패딩하는 패턴이 이미지 빌드/업그레이드 파일의 전형적 특징과 일치한다.
3. **두 파일의 코드 대부분(98.6%)은 동일**하지만, 나머지는 무작위 손상이 아니라 **서로 다른 펌웨어 빌드/버전 간의 정상적인 차이**로 해석하는 것이 가장 부합한다.
4. 따라서 **2026-07-22에 작성된 부트로더 stage2 정적 분석 보고서들은 레퍼런스 파일(또는 그 이전 소스) 기준으로 작성된 것이며, 그 자체로는 여전히 유효**하지만, `firmware_full.bin` 기준으로 stage2를 다시 분석하려면 압축 스트림 내용이 다를 수 있다는 점을 감안해야 한다.

## 7. 참고 — 재현에 사용한 명령

```python
import hashlib, zlib
ref = open('n602sr_ml_15_258.bin','rb').read()
new = open('firmware_full.bin','rb').read()

# 크기/해시
len(ref), len(new)
hashlib.sha256(ref).hexdigest(), hashlib.sha256(new).hexdigest()

# 바이트 단위 비교
minlen = min(len(ref), len(new))
diffs = [i for i in range(minlen) if ref[i] != new[i]]

# gzip 스트림 재현
zlib.decompressobj(47).decompress(ref[0x9a60:0x9a60+300000])   # 성공
zlib.decompressobj(47).decompress(new[0x9a60:0x9a60+300000])   # 실패 (invalid distance too far back)
```
